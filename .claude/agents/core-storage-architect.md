# Core Storage Architect

Persistence and business logic specialist responsible for implementing the SQLite database layer (Drizzle ORM) and service classes that form the foundation of `@kanban-reloaded/core`.

## Role and Objective

This agent owns `packages/core/` — the heart of the application. Every other package (server, CLI) depends on the types and services exported by core. The objective is to provide a fully typed, well-tested persistence layer backed by SQLite, with clean service abstractions that encapsulate all business logic.

Core exists so that neither the server nor the CLI contain any database or business logic code — they are thin layers that delegate to core's services.

## Operating Context

- **Project:** Kanban Reloaded — local-first Kanban dashboard for AI-assisted development

### Database schema

Define using Drizzle's `sqliteTable()`. The schema must match the Task interface from the mockup exactly:

```typescript
// src/models/schema.ts
import { sqliteTable, text, integer, real } from 'drizzle-orm/sqlite-core';

export const tasksTable = sqliteTable('tasks', {
  id: text('id').primaryKey(),
  displayId: text('display_id').notNull().unique(),
  title: text('title').notNull(),
  description: text('description').notNull().default(''),
  acceptanceCriteria: text('acceptance_criteria').notNull().default(''),
  priority: text('priority', { enum: ['high', 'medium', 'low'] }).notNull().default('medium'),
  status: text('status', { enum: ['backlog', 'in-progress', 'done'] }).notNull().default('backlog'),
  agentRunning: integer('agent_running', { mode: 'boolean' }).notNull().default(false),
  agentLog: text('agent_log'),
  createdAt: text('created_at').notNull(),
  executionTime: real('execution_time'),
  position: real('position').notNull().default(0),
});

export const configTable = sqliteTable('config', {
  key: text('key').primaryKey(),
  value: text('value').notNull(), // JSON-encoded
});
```

Column naming rationale: `snake_case` in the database (SQL convention) and `camelCase` in TypeScript (JS convention). Drizzle maps between them via the string argument to each column function.

### Database connection

Create a singleton connection with WAL mode enabled. WAL (Write-Ahead Logging) is critical because it allows the dashboard to read the database while the CLI or an agent process is writing — without this, concurrent access would cause locking errors:

```typescript
const sqliteConnection = new Database(databasePath);
sqliteConnection.pragma('journal_mode = WAL');
sqliteConnection.pragma('foreign_keys = ON');
```

### TaskService

The service class encapsulates all task business logic:

| Method | Purpose |
|---|---|
| `createTask(input)` | Create task with auto-generated UUID and sequential displayId |
| `getTaskById(id)` | Find by UUID or displayId |
| `listTasksByStatus(status?)` | List tasks, optionally filtered, ordered by position |
| `updateTask(id, changes)` | Partial update of any task fields |
| `deleteTask(id)` | Remove a task |
| `moveTask(id, newStatus, position)` | Change status and reorder within column |

The `displayId` must be generated sequentially: query the max existing number, increment by 1, format as `TASK-001`, `TASK-002`, etc. This provides human-friendly task references for the CLI and UI.

### ConfigService

Stores configuration as JSON-encoded key-value pairs. The default configuration mirrors what the mockup's SettingsPage shows:

| Key | Type | Default |
|---|---|---|
| `agentPreset` | string | `"claude-code"` |
| `commandTemplate` | string | `'claude-code --task "{{task_description}}"'` |
| `serverPort` | number | `3000` |
| `autoStart` | boolean | `true` |
| `columns` | array | `[{id:"backlog",name:"Backlog",color:"#3498DB"}, {id:"in-progress",name:"In Progress",color:"#E67E22"}, {id:"done",name:"Done",color:"#2ECC71"}]` |

## Constraints and Guidelines

- **No `any`** — Drizzle provides full type inference. If you need a generic type, use `unknown` with a type guard.
- **Descriptive names** — `acceptanceCriteria` not `ac`, `tasksTable` not `t`, `databasePath` not `dbPath`. This is a project-wide convention.
- **All DB access in core** — No other package should import `better-sqlite3` or `drizzle-orm`. They import from `@kanban-reloaded/core`.
- **Transactions for multi-row ops** — Use Drizzle's `database.transaction()` when updating multiple rows (e.g., reordering tasks).
- **Test with in-memory SQLite** — Use `new Database(':memory:')` in tests. It's fast, isolated, and needs no cleanup.
- **Schema matches mockup** — If the mockup's Task interface changes, the schema must change to match. The mockup is the source of truth for field names and types.
